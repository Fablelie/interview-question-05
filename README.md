# (interview-question-05)
# 🎟️ ระบบบริหารจัดการบัตรคิว

โปรเจกต์นี้เป็นระบบจำลองการบริหารจัดการคิวและออกบัตรคิวอัตโนมัติ (Queue Ticket System) ที่ออกแบบมาเพื่อรองรับการจัดคิวแบบหมุนวน (Running Sequence) ตั้งแต่หมายเลข **A0 ถึง Z9** และรองรับการสลับหน้าจอ 3 หน้าจอหลัก (IT 05-1, IT 05-2, IT 05-3) ตามเงื่อนไขของโจทย์

ในส่วนของระบบหลังบ้าน ได้เลือกประยุกต์ใช้แนวคิด **Clean Architecture** ร่วมกับกลไก **Optimistic Locking** ในการจัดการฐานข้อมูล เพื่อช่วยดูแลความถูกต้องของลำดับคิว และลดโอกาสการเกิดหมายเลขคิวซ้ำซ้อนในกรณีที่มีการส่งคำขอเข้ามาพร้อมกันครับ

---

## 🛠️ เทคโนโลยีที่เลือกใช้ (Tech Stack)

*   **Frontend:** Angular
*   **Backend:** Golang (Fiber v3 Framework)
*   **Database:** Microsoft SQL Server 2022
*   **Database Management:** Adminer (Web-based Database Client สำหรับช่วยเปิดดูตาราง)
*   **Infrastructure:** Docker & Docker Compose
*   **Testing Tool:** Postman (สำหรับใช้ทดสอบการทำงานของระบบ)

---

## 🖥️ รายละเอียดหน้าจอและการทำงาน (UI Flows)

1.  **หน้าหลักรับบัตรคิว (IT 05-1):** 
    *   ประกอบด้วยปุ่ม **"รับบัตรคิว"** และ **"ล้างคิว"**
    *   เมื่อกดปุ่ม "รับบัตรคิว" ระบบจะส่งคำขอไปยัง API หลังบ้านเพื่อดึงเลขคิวถัดไป (A0 - Z9) และนำทางไปยังหน้า **IT 05-2**
    *   เมื่อกดปุ่ม "ล้างคิว" ระบบจะนำทางไปยังหน้า **IT 05-3** โดยในสเต็ปนี้จะยังไม่มีการล้างข้อมูลในฐานข้อมูลจริง เพื่อให้ผู้ใช้ได้ตรวจสอบสถานะก่อนครับ
2.  **หน้าแสดงหมายเลขบัตรคิว (IT 05-2):**
    *   แสดงหมายเลขบัตรคิวล่าสุดที่ระบบอนุมัติ พร้อมระบุ วัน/เดือน/ปี และเวลาสากล (UTC) ที่บันทึกบัตรคิวใบดังกล่าว
    *   มีปุ่ม **"กลับไปหน้ารับบัตรคิว"** เพื่อย้อนกลับไปหน้าหลัก (IT 05-1)
3.  **หน้าจัดการล้างคิวระบบ (IT 05-3):**
    *   เมื่อเข้าสู่หน้านี้ครั้งแรก ระบบจะแสดง **หมายเลขคิวปัจจุบันล่าสุด** ที่ดึงมาจากฐานข้อมูล เพื่อให้ผู้ใช้ทราบสถานะปัจจุบันก่อนทำการรีเซ็ต
    *   มีปุ่ม **"เริ่มต้นทำการล้างคิว"** เมื่อกดปุ่มนี้ ระบบจะส่งคำสั่งไปรีเซ็ตค่าตัวนับคิวในฐานข้อมูล และปรับตัวเลขบนหน้าจอให้กลายเป็น **`00`** เพื่อยืนยันว่าทำการล้างคิวเรียบร้อยแล้ว
    *   มีปุ่ม **"กลับไปหน้ารับบัตรคิว"** เพื่อย้อนกลับไปหน้าหลัก โดยการกดรับบัตรคิวในรอบถัดไปจะกลับไปเริ่มต้นที่หมายเลข **`A0`** ใหม่เสมอครับ

## ⚡ วิธีการเปิดรันระบบ (How to Run)

ตัวระบบถูกมัดรวมไว้ใน Docker เรียบร้อยแล้ว เครื่องคอมพิวเตอร์ที่ใช้ตรวจงานจึงจำเป็นต้องมีโปรแกรม **Docker Desktop** เปิดสแตนด์บายไว้ จากนั้นสามารถเลือกเปิดรันระบบผ่านไฟล์สคริปต์ตามระบบปฏิบัติการได้เลยครับ:

### 🪟 สำหรับผู้ใช้ Windows:
*   **เปิดระบบ:** ดับเบิ้ลคลิกที่ไฟล์ `run-windows.bat` เพื่อสั่งให้ Docker เริ่มประกอบร่างและเปิดบริการต่าง ๆ
*   **หยุดและล้างข้อมูล:** ดับเบิ้ลคลิกที่ไฟล์ `clean-window.bat` เมื่อต้องการหยุดการทำงานและเคลียร์ข้อมูลใน Volume ทิ้งครับ

### 🍏 สำหรับผู้ใช้ macOS / Linux:
*เนื่องจากระบบรักษาความปลอดภัยของ Mac ในครั้งแรกจำเป็นต้องเปิด Terminal ในโฟลเดอร์นี้แล้วพิมพ์คำสั่งปลดล็อกสิทธิ์ไฟล์ก่อนครับ:* `chmod +x run-mac.sh clean-mac.sh`
*   **เปิดระบบ:** ดับเบิ้ลคลิกที่ไฟล์ `run-mac.sh`
*   **หยุดและล้างข้อมูล:** ดับเบิ้ลคลิกที่ไฟล์ `clean-mac.sh`

## 🗄️ การจัดการฐานข้อมูล (Database & Auto-Migration)

## 🚀 การทดสอบกรณีส่งคำขอพร้อมกัน (Postman Performance Test)

เพื่อตรวจสอบการทำงานของระบบป้องกันคิวซ้ำ (Concurrency Control) ตามเงื่อนไขของโจทย์ เราสามารถใช้ฟังก์ชัน Performance Test ของ Postman ในการจำลองสถานการณ์ได้ตามขั้นตอนดังนี้ครับ:

### 📥 ขั้นตอนที่ 1: การนำเข้าคำขอ (Import Collection)
1. เปิดโปรแกรม **Postman**
2. กดปุ่ม **Import** ที่มุมซ้ายบน แล้วเลือกไฟล์ `Queue-Ticket-System.postman_collection.json` จากโฟลเดอร์นอกสุดของโปรเจกต์นี้เข้าสู่โปรแกรม

### 🔥 ขั้นตอนที่ 2: ตั้งค่าจำลอง Load Testing
1. คลิกปุ่ม **`...` (สามจุด)** หลังชื่อ Collection ที่เพิ่ง Import เข้ามา แล้วเลือกเมนู **Run collection**
2. ที่เมนูด้านบน ให้เลือกแท็บ **`Performance`**เพื่อสลับเข้าสู่โหมดจำลองคำขอพร้อมกัน
3. ตั้งค่าพารามิเตอร์เพื่อใช้ทดสอบระบบ:
   * **Virtual Users (VU):** ระบุเป็น `20` หรือเพิ่มขึ้นตามต้องการ (จำลองจำนวนผู้ใช้ที่ส่งคำขอกดคิวพร้อมกันในเสี้ยววินาที)
   * **Duration:** ระบุเป็น `1` นาที (ระยะเวลาในการสุ่มยิงทดสอบต่อเนื่อง)
   * **Load Profile:** เลือกเป็น `Fixed` เพื่อคงระดับคำขอไว้ตลอดการทดสอบ
4. กดปุ่มสีส้ม **Run** เพื่อเริ่มต้นกระบวนการทดสอบครับ

### 📊 ขั้นตอนที่ 3: การตีความผลลัพธ์
1. **ผลลัพธ์ใน Postman:** เนื่องจากตัวระบบมีการวางระบบป้องกันไว้ 2 ชั้น (**Double-Lock Mechanism**) คือตัวแปรล็อกในระดับแอปพลิเคชัน `uc.mu.Lock()` ควบคู่กับเลขเวอร์ชัน `VersionLock` ในระดับฐานข้อมูล ระบบจะช่วยจัดระเบียบคำขอที่เข้ามาพร้อมกันให้ประมวลผลต่อคิวกันอย่างเป็นระบบ ส่งผลให้อัตราความผิดพลาด (Error Rate) มีค่าเป็น **0%** คิวสามารถรันต่อไปได้อย่างราบรื่นและไม่เกิดอาการระบบหยุดทำงานครับ
2. **ผลลัพธ์ในฐานข้อมูล:** เมื่อล็อกอินเข้าไปตรวจสอบประวัติตั๋วในตาราง `QueueTickets` ผ่าน Adminer จะพบว่าหมายเลขบัตรคิวถูกบันทึกเรียงลำดับตัวอักษรและตัวเลขเพิ่มขึ้นอย่างถูกต้อง และ**ไม่มีหมายเลขคิวใดบันทึกซ้ำซ้อนกันในฐานข้อมูล** ซึ่งเป็นไปตามเป้าหมายในการรักษาความถูกต้องของข้อมูลคิวตามที่โจทย์กำหนดไว้ครับ
